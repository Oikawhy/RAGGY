## Архитектурные решения (Architectural Decisions)

- Почему pgvector? для тестовой задачи требуется transactional data model (sources - sections - chunks с metadata, versioning, deduplication) и hybrid retrieval (vector + lexical). PostgreSQL закрывает обе задачи в одной системе. Vector search — один шаг из шести в pipeline; Qdrant/Pinecone решают только его, а metadata, FTS, versioning и deduplication всё равно требуют реляционную БД. Два хранилища удваивают инфраструктурную сложность ( избыточны для тестов) без пропорциональной выгоды.
- почему  in-memory BM25? та же причина по сути,  так как это второй этап после FTS, при переходе необходимо выносить в отдельный сервис.  я бы выбрал OpenSearch для нормального BM25
- почему BGE-M3? Хорошая мультиязычная модель (по сравнению с е5 субъективно украинский лучше понимает), локалочка, точно также нужно выносить в отдельный сервис  чтобы воркеры по апишке дёргали
- reranker я бы взял тоже от BGE  (сейчас реализован простой лексический)
- по метадате: храним источник, секцию, чанк как отдельные сущности вместе с метадатой, это понадобится для будущего: фильтрации, версионирования, аудита, rbac





## Интеграция с Laravel
Честно, линейный RAG не подходит для интеграции в вашу систему, учитывая, что у вас уже используется агентный стек и есть жесткая юридическая специфика. Для реального сервиса систему нужно сразу проектировать как агентную.
Агентная архитектура лучше обычного линейного RAG, потому что юридический вопрос редко сводится к простому retrieve - generate. Система должна уметь планировать поиск, проверять права доступа, искать по нескольким источникам, оценивать релевантность, актуальность документов, проверять ссылки и контролировать галлюцинации.
В такой архитектуре каждый агент отвечает за отдельный этап: маршрутизацию запроса, построение плана, retrieval, reranking, проверку источников, генерацию ответа, citation verification и risk escalation. Это делает ответы более надежными, трассируемыми и безопасными для legal-domain, особенно когда есть конфликт источников, недостаток фактов


## запуск


### 1. Подготовка окружения
Создайте файл настроек из шаблона:
```bash
cp .env.example .env
```
Откройте созданный файл `.env` и укажите ваш ключ `OPENAI_API_KEY`. Также убедитесь, что установлен правильный бэкенд для эмбеддингов:
```text
EMBEDDING_BACKEND=local_bge_m3
```

### 2. Установка зависимостей
Убедитесь, что у вас установлен Python 3.12. Выполните команды:
```bash
python3.12 -m venv venv
source venv/bin/activate
pip install -r requirements.txt
```

### 3. Запуск баз данных
Запустите PostgreSQL и Redis через Docker Compose:
```bash
docker compose up -d postgres redis
```

### 4. Импорт базы знаний (Индексация)
Загрузите и обработайте текст базы знаний:
```bash
python scripts/ingest_kb.py \
  --source data/knowledge_base.md \
  --logical-source-key assignment-kb \
  --version 1 \
  --apply \
  --embed
```

### 5. Запуск веб-сервиса
Запустите сервер разработки:
```bash
python -m uvicorn app.main:create_production_app --factory --reload --port 8765
```

Также доступен альтернативный вариант быстрого запуска всего проекта в Docker:
```bash
docker compose up --build
```

## Подход к индексации и поиску

### Индексация (Indexing)
Процесс импорта полностью отделен от обработки запросов пользователей. Скрипт разбивает файл `knowledge_base.md` на разделы и текстовые блоки (чанки), рассчитывает уникальные хэши для предотвращения дубликатов, генерирует векторные представления с помощью модели `BGE-M3` и записывает все это в PostgreSQL.

### Поиск (Retrieval)
Используется гибридный поиск (**hybrid retrieval**), сочетающий три компонента:
1. **Векторный поиск:** через расширение `pgvector` (индекс HNSW) ищет смысловое соответствие запроса.
2. **Лексический поиск:** с помощью встроенного полнотекстового поиска PostgreSQL FTS находит точные совпадения слов.
3. **BM25 поиск:** быстрый in-memory алгоритм для точного текстового поиска на базе активных документов.

Результаты этих трех видов поиска объединяются с помощью алгоритма **RRF**, после чего проходят через быстрый переранжировщик (**reranker**) и расширяются соседними абзацами для лучшего контекста перед отправкой в LLM.

## Что логируется в trace

Каждый запрос пользователя записывается в виде одной строки JSON в общий файл `traces/traces.jsonl`. Лог содержит детальную информацию о каждом шаге:
- Уникальный `trace_id` запроса.
- Текст вопроса пользователя.
- Детальное описание и длительность (в миллисекундах) каждого шага: время генерации эмбеддинга, поиска в БД, переранжирования, работы LLM.
- Найденные источники и отправленный в модель контекст.
- Результат проверки надежности и причины срабатывания fallback.
- Финальный ответ системы и общая длительность запроса.

---

## Как определяется confidence

Уровень уверенности системы (`high`, `medium` или `low`) рассчитывается автоматически на основе нескольких объективных факторов:
- **Оценка поиска (RRF score):** насколько качественные документы удалось найти.
- **Согласованность алгоритмов:** совпадают ли результаты векторного поиска с лексическим (BM25).
- **Количество найденных блоков:** достаточно ли найдено подтверждений в базе знаний.
- **Маркеры неуверенности:** содержит ли найденный контекст оговорки ("информация отсутствует", "не описано") и сообщила ли сама LLM о нехватке данных в тексте своего ответа.

---

## Когда срабатывает fallback

Сервис автоматически переходит в режим безопасного отказа (fallback) и заполняет поле `fallback_reason`, если:
1. **Нет контекста:** поисковые алгоритмы не нашли ни одного релевантного абзаца в базе знаний.
2. **Сработала защита от выдумывания расчетов:** запрос требует вычислений (например, "посчитай сумму"), но в найденных документах нет числовых данных или формул (индексов инфляции, базового месяца и т.д.).
3. **LLM сигнализирует о нехватке данных:** языковая модель прямо ответила, что в предоставленном контексте ответа на вопрос нет.
4. **Ошибка API или системы:** произошел сбой при обращении к OpenAI или БД.
5. **Лимиты запросов:** превышена настроенная частота запросов (rate limiting).

Во всех этих случаях пользователь получит четкий и вежливый ответ о том, что информация отсутствует, вместо вымышленных фактов.

---

## Какой LLM API использован

Генерация ответов выполняется через официальное API **OpenAI API** с использованием структурированного вывода (JSON schema).
- **Модель по умолчанию:** `gpt-5.4-mini` (настраивается через параметр `LLM_MODEL`).
- Система требует от модели строго ограниченный формат ответа в виде JSON с тремя полями: `answer` , `fallback_reason`  и `used_sections` 
- Запрос описывает поведение, но контракт ответа обеспечивается ограничениями схемы на уровне API и валидацией Pydantic.
